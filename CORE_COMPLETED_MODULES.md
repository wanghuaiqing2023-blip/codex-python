## Status Rules

- `complete`: the listed Rust module or behavior contract has Python implementation plus focused validation. This may still be narrower than whole-crate parity, but it should be treated as a full module/task blocker unless Rust upstream changed, a test fails, or the user asks for that module.
- `slice complete`: a named behavior slice is complete, but the whole Rust module is not. Avoid redoing that slice; continue only from a listed deferred boundary.
- `complete shim`: a compatibility shim is validated for current core use, but the full Rust module behavior is intentionally not implemented.
- `complete enough`: no current core gap is known after source review and focused tests, but the module should be rechecked before using it as whole-module parity evidence.
- `pending validation`: there is recorded implementation/parity work, but the status explicitly says validation is pending or incomplete. Do not treat as fully complete.
- `utility complete`: a non-core utility crate used by `codex-core` is complete enough to avoid re-porting during core work.
